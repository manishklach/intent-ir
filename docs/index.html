INTENT-IR: Verifiable Execution for AI Agents

An assembly-style communication layer where agents send instruction packets that can be disassembled, verified, rejected, executed, traced, and replayed.

Core idea: agents should not blindly execute requests. They should execute verified programs.

The problem

Most agent systems treat tool calls as trusted requests. An agent emits JSON, a runtime invokes a tool, and logs are inspected after the fact.

That model is convenient, but it is weak for safety and debugging. Before execution, the receiver often cannot cleanly answer: what exactly will run, is it allowed, what budget does it claim, and can the result be replayed?

Opaque
Tool calls are readable, but not a stable execution format.
Hard to verify
Policy is usually scattered across runtime code.
Hard to replay
Logs are not the same as deterministic execution traces.

The model

Agent Message JSON
    ↓ compile-message
IntentASM (.intentasm)
    ↓ asm
IntentBin (.intentbin)
    ↓ recv
Disassemble + Verify(policy)
    ↓
Execute OR Reject
    ↓
Trace → Replay

INTENT-IR treats agent communication like a compiler/runtime pipeline. A high-level agent message is lowered into a small assembly-like program, assembled into a binary packet, verified by the receiving agent, and only then executed.

IntentASM example

AGENT planner
AGENT worker

TASK t1 TYPE "repo_scan"
BUDGET t1 MEMORY 512MB TIME 30000MS TOKENS 4096

PAYLOAD p1 JSON {"repo": ".", "include": ["*.md", "*.py"]}

SEND planner worker CHANNEL "task" PAYLOAD p1
CALL worker TOOL "repo.scan" ARGS p1

ASSERT LAST.status == "ok"
COMMIT t1 ARTIFACT "repo_summary.json"
HALT

This is the key difference: the sender is not just asking the receiver to do something. It is sending a small, inspectable program.

Killer demo: safe vs unsafe packets

The demo shows two packets sent to the same worker agent. One is safe and executes. One attempts an unsafe tool call and is rejected before execution.

$ make demo-safety

SAFE PACKET:
[policy] loaded policies/worker.policy.json
[verify] ✔ passed
[execute] CALL repo.scan
[commit] repo_scan.report
[trace] wrote traces/safe_repo_scan.intenttrace.jsonl
[replay] ✔ success

UNSAFE PACKET:
[policy] loaded policies/worker.policy.json
[verify] ✖ failed
[reject] tool "shell.exec" not allowed
[execute] skipped
[trace] wrote traces/unsafe_shell.intenttrace.jsonl
[replay] ✔ confirms rejection
The important property: the unsafe packet does not run. The receiver rejects it at the verification gate.

Policy enforcement

Each receiver agent can enforce its own policy. The worker may allow repository scanning and artifact writing, while denying shell execution, network calls, deletion, or secrets access.

{
  "allowed_tools": ["repo.scan", "artifact.write", "trace.emit"],
  "denied_tools": ["shell.exec", "file.delete", "network.post", "secrets.read"],
  "max_budget": {
    "memory_mb": 512,
    "tokens": 5000,
    "wall_ms": 5000
  },
  "require_asserts": true,
  "require_commit": true
}

The verifier checks the packet structure, tool permissions, resource budget, and required invariants before the runtime executes anything.

Systems analogy

The idea is closer to eBPF or WASM than to normal tool-calling. eBPF programs are verified before being loaded into the kernel. WASM modules execute inside a constrained runtime. INTENT-IR applies the same instinct to agent communication: instruction packets should be inspectable, policy-checked, and replayable.

This is not a production agent runtime. It is a research prototype showing a direction: agent systems need a verifiable control plane.
